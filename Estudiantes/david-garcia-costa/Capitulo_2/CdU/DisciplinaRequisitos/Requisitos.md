# Requisitos funcionales

## RF01. Incorporación de documentación

El sistema debe permitir incorporar documentación de entrada de tipo DRF, DDS.

## RF02. Asociación de documentación a proyecto

El sistema debe permitir asociar la documentación incorporada a un proyecto, de forma explícita o inferida durante el flujo de trabajo.

## RF03. Extracción de casos de uso

Se debe identificar y estructurar los casos de uso contenidos en la documentación de entrada.

## RF04. Extracción de requisitos funcionales

Se debe identificar y estructurar los requisitos funcionales contenidos en la documentación de entrada.

## RF05. Gestión de casos de uso

Se debe permitir crear, consultar, modificar y eliminar casos de uso.

## RF06. Gestión de requisitos funcionales

Se debe permitir crear, consultar, modificar y eliminar requisitos funcionales.

## RF07. Generación de escenarios de prueba

Se debe generar escenarios de prueba en formato Gherkin a partir de los casos de uso y requisitos funcionales.

## RF08. Gestión de escenarios de prueba

Se debe permitir crear, consultar, modificar, eliminar y versionar escenarios de prueba.

## RF09. Gestión de feedback sobre escenarios

Se debe permitir incorporar feedback sobre escenarios de prueba para su iteración antes de publicación.

## RF10. Publicación de casos de prueba

Se deben agrupar los escenarios aprobados en casos de prueba y publicarlos en Kiwi TCMS.

## RF11. Gestión de casos de prueba publicados

Se debe permitir consultar, modificar y eliminar casos de prueba publicados en Kiwi TCMS.

## RF12. Búsqueda de casos de prueba

Se debe permitir buscar casos de prueba en Kiwi TCMS mediante distintos criterios, como resumen, texto, categoría, prioridad, estado, usuarios, etiquetas y automatización.

## RF13. Trazabilidad

El sistema debe mantener trazabilidad entre proyecto, documentación, casos de uso, requisitos funcionales, escenarios de prueba y casos de prueba.

---

# Requisitos no funcionales
## RNF01. Consistencia

El sistema debe mantener coherencia entre los artefactos generados durante el flujo de transformación de requisitos a casos de pruebas.

## RNF02. Trazabilidad

El sistema debe conservar referencias de origen entre documentación, proyecto, escenarios y casos de prueba publicados.

## RNF03. Usabilidad

El sistema debe ofrecer operaciones claras y diferenciadas para la consulta, modificación, eliminación y publicación de artefactos.

## RNF04. Extensibilidad

El sistema debe permitir incorporar nuevas fuentes documentales, nuevos criterios de transformación y nuevas operaciones de gestión sin alterar su estructura general.

RNF05. Integración

El sistema debe integrarse con Kiwi TCMS para la publicación, consulta y gestión de casos de prueba.

## RNF06. Mantenibilidad

El sistema debe estar organizado en módulos funcionales diferenciados para facilitar su evolución y mantenimiento.


## RNF07. Identificación de contexto

El sistema debe preservar la referencia al proyecto asociado a cada artefacto relevante.
